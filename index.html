OpenStreetMap Data Case Study
The goal of this project is to obtain map data from OpenStreetMap and review for potential errors.
I chose to examine a town in Japan named Niseko because I was interested in visiting the area for
snowboarding and wanted to learn more about it. I went to the OpenStreetMap website and selected
the boundary of the town. Then, I downloaded the OSM file for analysis.
https://www.openstreetmap.org/relation/4087840#map=11/42.7927/140.6846
Problems Encountered in the Map
I took a small sample from the data file to make it easier for auditing. After performing an audit of the
sample file, I found three problems with the data.
Unicode characters for English names (i.e Japanese characters)
Unicode characters in postal code (i.e Japanese postal symbol)
Phone numbers in multiple formats (i.e with country code vs without country code)
Unicode characters for English names
I browsed through the tag attributes and saw one instance where the value of "name:en" key was in
unicode characters. I believe that the "name:en" key refers to the English name of a place. I also saw
a "name:jp" key, so there seems to be a distinction between names in different languages. I chose to
audit "name:ja" keys to make sure that their values were all unicode characters, and that "name:en"
keys were alphanumeric. First, I examined the values of "name:en" keys to see if there were any
unicode characters because I didn't think that plain English names would require any unicode
characters. The results are shown below. The first row shows the unicode characters, and the second
row shows the same name with unicode escape characters.
\u30ea\u30ba\u30e0\u30b5\u30a4\u30af\u30eb
Takahashistore
Takahashi\u3000store
Kygoku Onsen
Ky\u014dgoku Onsen
There were only three items that contained unicode characters. Only the first result was in Japanese
characters and needed to be fixed. The second result only had an unicode space character. There is
no downside to replacing the space character with an empty string, but I couldn't think of a benefit
either, so I left it alone. Unicode was needed to display a special character in the third result, so I also
left it untouched. I searched the characters in Google and found out that the correct English name is
"Rhythm Cycles".
In order to fix the first result, I replaced the value for that key with the correct English name. I also
checked to see if there were any Japanese names that weren't in unicode, and there were none.
Unicode characters in postal code
The postal codes seemed to be either 7 digits in a row, or separated by a dash after the 3rd digit. I
wanted to make them all have a dash so they would look more uniform. Also, there was one postal
code with an unicode character representing the Japanese postal symbol. I wanted to remove it as I
don't think the symbol was necessary.
def audit_postal(tag):
'''Remove postal symbol and return postal codes with dashes'''
allnum = re.compile(r'\d{7}')
if is_post_code(tag):
if allnum.search(tag.attrib['v']):
digits = allnum.search(tag.attrib['v'])
grouped_num = digits.group(0)[0:3] + "-" + digits.group(0)[3:]
tag.attrib['v']=grouped_num
if isinstance(tag.attrib['v'],unicode):
remove_uni = tag.attrib['v'][1:].encode("utf-8")
tag.attrib['v']=remove_uni
return tag

According to my research, area codes in Japan varies in length, so it's too difficult to set a pre-
determined length. They were also either embedded into the beginning of the phone number, or
separated by a dash before the phone number. 050,080,and 090 are area codes for mobile phone
numbers. Also, leading zeros are not included when calling within Japan. I thought that it would be
less confusing if the leading zero was preserved for all phone numbers as some had leading zeroes
while others didn't. I also got rid of extra parentheses, empty spaces, and dashes aside from the first
dash separating the country code +81. If the country code wasn't already included, then I added it in
an attempt to standardize the phone numbers.

Additional Improvement Suggestion
I decided to browse through some of the other items available and across a few restaurants. The
information provided ranged from very basic (only the name) to type of cuisine and operating hours. It
would be nice if things like operating hours and phone numbers were added for all restaurants. It
would be even better if users could add a quality rating (1 - 5 stars) and/or a price rating ($ -
$$$$$). I think that it would be helpful for travellers and locals alike.
Anticipated Problems
It may be difficult to gather the information needed to implement my suggestions above. Some of the
smaller restaurants may not have information such as phone numbers available. It would require
users in the local area to contact the places and gather the information if available. The places might
not even want more information to be displayed on the internet than just the name.

Conclusion
After reviewing the data, I felt like the data could use more cleaning to ensure that the formatting of
information is displayed more consistently. Also, it would make the information easier to read and
cause less confusion to the user when some items of the same type (such as restaurants) have alot
of information while other similar items have a lot less information.
